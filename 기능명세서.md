## 0. 공통 사항

### 0.1 인증
- OAuth2 소셜 로그인(Google/Kakao) → JWT 발급
  - Access 20분 (`Authorization: Bearer`), Refresh 14일
- 보호 API는 `JwtAuthenticationFilter` 검증, 실패 401.

### 0.2 공통 응답 포맷
```jsonc
{ "success": true,  "data": { /* ... */ }, "error": null }
{ "success": false, "data": null, "error": { "code": "STRING", "message": "..." } }
```

### 0.3 공통 예외 코드
| HTTP | code | 설명 |
|------|------|------|
| 400 | `INVALID_PARAMETER` | 검증 실패 |
| 401 | `UNAUTHORIZED` | 인증 누락/만료 |
| 403 | `FORBIDDEN` | 권한 없음(모임 비소속 등) |
| 404 | `NOT_FOUND` | 리소스 없음 |
| 409 | `CONFLICT` | 무결성 위반(XOR, 중복) |
| 413 | `PAYLOAD_TOO_LARGE` | 업로드 용량 초과 |
| 500 | `INTERNAL_ERROR` | 서버 오류 |

## 1. 소셜 로그인 (S1, S2 · FR-AUTH) — *기존 구현 기반*

### 1.1 개요
구글/카카오 OAuth2로 로그인하고 JWT를 발급받는다. 최초 로그인 시 회원 자동 생성. (현재 `SecurityConfig`, `CustomOAuth2UserService`, `OAuth2SuccessHandler`, `JwtProvider` 구현됨)

### 1.2 화면
- **S1 랜딩**: Hero + 3카드(자동 일정/공동 편집/관광지 검색) + "시작하기"
- **S2 로그인**: "구글로 시작하기", "카카오로 시작하기" (※ 와이어프레임의 이메일/비번 폼은 폐기 — CON-02)

### 1.3 흐름
```
[소셜 버튼] → OAuth2 동의 → CustomOAuth2UserService(회원 조회/생성)
  → OAuth2SuccessHandler(JWT 발급) → 프론트 토큰 전달 → 인증 완료 → S3 이동
```

### 1.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/oauth2/authorization/{google\|kakao}` | 소셜 로그인 시작 | - |
| GET | `/login/oauth2/code/{provider}` | 콜백(내부) | - |
| POST | `/api/auth/refresh` | Access 재발급 | refresh |
| POST | `/api/auth/logout` | 로그아웃 | ✅ |
| GET | `/api/auth/me` | 내 정보 | ✅ |

### 1.5 데이터 (`users`)
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | BIGINT PK | |
| email | VARCHAR UNIQUE | |
| name | VARCHAR | |
| profile_image_url | VARCHAR | |
| role | VARCHAR | ADMIN/USER |
| created_at | DATE | |

### 1.6 예외
- 이메일 미동의 → 400 / 토큰 만료 → 401(→refresh) / refresh 무효 → 재로그인

## 4. 여행 계획 자동 생성 (S5 · FR-PLAN-01,02) ⭐

### 4.1 개요
지역/기간/인원/**스타일**을 입력하면 TourAPI 관광지를 기반으로 날짜·시간별 일정을 자동 추천한다. "스타일"은 콘텐츠 타입 선호로 매핑(CON-05): 예) 자연→관광지, 맛집→음식점, 문화→문화시설.

### 4.2 화면
- 입력 폼: 지역(시/도, 구/군), 출발일·종료일, 인원, **여행 스타일(다중 선택)**
- "자동 생성 ✨" → 결과는 S6(편집 화면)으로 이동

### 4.3 자동 생성 로직(초안)
```
입력: region, startDate, endDate, headcount, styles[]
1) 후보 = attractions WHERE 지역 일치 AND contentTypeId ∈ map(styles)
2) 정렬 = (이미지 유무, 대표성, 콘텐츠타입 매칭도)
3) 일자 분배 = 기간 일수만큼 Day 분할, Day당 N개(예: 3~4) 배치
4) 시간 슬롯 = 오전/점심(음식점 우선)/오후/저녁 규칙 배치
5) trips 생성 + data.items[](visitDate·order·time) 채움
```
> 알고리즘 고도화(동선 최적화·거리 기반)는 후속. MVP는 규칙 기반.

### 4.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| POST | `/api/plans/auto` | 조건 입력→일정 자동 생성 | ✅ |

**요청 예**
```jsonc
{ "sidoCode":1, "gugunCode":1, "startDate":"2026-06-01", "endDate":"2026-06-03",
  "headcount":2, "styles":["자연","도보"], "groupId": 10 }   // groupId 없으면 개인
```

### 4.5 데이터 & 트리거
- 저장: `trips`(`data` JSONB에 items[] 채움). 행 테이블(trip_attractions)·스냅샷 테이블 없음.
- `updated_at` 트리거로 갱신 시각 관리.

### 4.6 예외
- 후보 부족 → 가능한 만큼 생성 + `partial:true` 안내
- endDate<startDate → 400 / 모임 비소속 groupId → 403 / XOR 위반 → 409

## 5. 여행 계획 편집 + 실시간 공동 편집 (S6 · FR-PLAN-03~05, FR-COL) ⭐

### 5.1 개요
일자별 일정을 조회·편집하고, **모임원이 동시에 실시간 공동 편집**한다. 편집은 WebSocket으로 송신, **Redis Pub/Sub로 서버 간 fan-out**되어 모든 참가자에게 즉시 전파된다(수평 확장 지원).

### 5.2 화면
- 헤더: 여행 제목/기간/인원 + **참가자 아바타** + 🔗공유 + ⋯
- 본문: Day별 타임라인(시간·관광지), 항목별 "○○ 편집중…" 표시, "+ 일정 추가"
- 우하단: 🟢 실시간 동기화 상태

### 5.3 REST API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/api/trips/{tripId}` | 여행 상세(`data.items` 포함) | ✅ 멤버 |
| PATCH | `/api/trips/{tripId}` | 제목/기간/**data 부분 갱신**(비실시간 저장 경로) | ✅ 멤버 |
| DELETE | `/api/trips/{tripId}` | 여행 삭제 | ✅ 소유/멤버 |

> **실시간 항목 편집**(추가/수정/삭제/순서)은 행 REST가 아니라 **§5.4 WebSocket `EDIT_*` 이벤트**로 처리 → Redis → Batch Worker가 `trips.data`를 통째 flush(JSONB upsert). 문서모델 전환으로 행 단위 엔드포인트(`/attractions/{id}`)는 제거.

### 5.4 실시간 채널 (WebSocket + Redis)
| 종류 | 경로 | 설명 |
|------|------|------|
| WS connect | `/ws/plans/{tripId}` | 편집 세션 입장(인증·멤버 검증) |
| 송신 이벤트 | `EDIT_ADD / EDIT_UPDATE / EDIT_DELETE / EDIT_REORDER / PRESENCE` | 편집 동작 |
| Redis Pub/Sub | `plan:{tripId}:edit` | 서버 간 fan-out |
| Redis(상태) | `plan:{tripId}:state` | 현재 일정 스냅샷(신규 참가자 즉시 로드) |

**동기화 흐름 (아키텍처 다이어그램 ① ~ ⑥)**
```
편집 → ① 서버가 Redis에 상태/이벤트 쓰기 → ② PUBLISH plan:{id}:edit
     → ③ 각 서버 SUBSCRIBE → 연결된 클라이언트로 push
신규 참가 → ④ Redis state 조회로 현재 일정 즉시 로드
Batch Worker → ⑤ Pull → ⑥ trips.data 영속화(JSONB upsert, 정합성 보정)
```

### 5.5 동시 편집 정책(MVP)
- 항목 단위 last-write-wins. 충돌 판정은 **Redis seq/Stream 순번**이 담당 → DB `version` 컬럼 불필요(ERD §4).
- 고급 CRDT/OT는 후속

### 5.6 예외
- 멤버 아님 → 403 / 삭제된 항목 편집 → 409 `CONFLICT` / 연결 끊김 → 재연결 후 state 재동기화

## 7. 멀티미디어 발자취 지도 (S8 · FR-MAP) ⭐ 핵심

### 7.1 개요
그룹이 **다녀온(end_date<오늘)** 여행에서 남긴 **사진·음성·영상**을 전국 지도에 발자취로 누적 표시한다. 미디어 본체는 각 여행의 `trips.data` items[].media에 있고, **지도는 `group_region_media`(지역당 대표 추억 1개)** 를 소스로 한다. **권역 색칠은 프론트가 GeoJSON 권역 폴리곤을 `group_region_media` 보유 여부로 칠한다(서버 choropleth 집계 없음).**

### 7.2 화면
1. **발자취 지도**: 그룹이 다녀온 시도/구군 권역 색칠(프론트 GeoJSON) + **지역당 대표 추억 1개 핀**
2. **드릴다운**: 권역/핀 클릭 → 그 지역 여행들의 미디어 갤러리
3. **미디어 팝업**: 사진 갤러리 / 음성 플레이어 / 영상 플레이어
4. **업로드**: 여행 편집에서 항목(블록)에 미디어 업로드(사진 우선 → 음성 → 영상)
5. **대표 큐레이션**: 지역 미디어 중 하나를 그룹 지도 대표로 지정

### 7.3 흐름
```
group_id
 → group_region_media WHERE group_id           // 지역별 대표 1개 + geom(핀)
   → 프론트: 권역 GeoJSON 색칠 + 대표 핀 렌더
 → (드릴다운) 그 지역 trips(end_date<today).data items[].media 집계 → 갤러리
```

### 7.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/api/groups/{groupId}/map` | 발자취 지도 소스 = group_region_media(지역별 대표+geom) | ✅ 멤버 |
| GET | `/api/groups/{groupId}/map?sidoCode=&gugunCode=` | 그 지역 여행들의 미디어 갤러리(data.items 집계) | ✅ 멤버 |
| POST | `/api/trips/{tripId}/media` | 미디어 업로드(멀티파트, `blockId`) → 스토리지 → 해당 item.media[] append | ✅ 멤버 |
| POST | `/api/groups/{groupId}/map` | 대표 추억 큐레이션 → group_region_media upsert(지역당 1개) | ✅ 멤버 |
| DELETE | `/api/groups/{groupId}/map/{id}` | 지도 대표 제거(미디어 본체는 항목 편집으로) | ✅ 멤버 |

**응답 예 (`/map`)**
```jsonc
{ "success": true, "data": [
  { "id":1, "sidoCode":26, "gugunCode":1, "mediaType":"PHOTO",
    "mediaUrl":"https://...a.jpg", "lat":35.158, "lng":129.16, "tripId":10 }
], "error": null }
```
> 권역 색칠 강도는 서버가 집계하지 않는다 — 프론트가 GeoJSON 권역에 **대표 보유 여부**로 칠한다.

**업로드 (`multipart/form-data`)**: `file`(바이너리), `blockId`(대상 item id), `mediaType`(PHOTO|AUDIO|VIDEO), `metadata`(JSON, 선택)
→ 서버: 스토리지 업로드 → URL/썸네일 → 해당 `trips.data` item의 `media[]`에 append(JSONB)

### 7.5 데이터 — 미디어 저장 위치
- **미디어 본체**: `trips.data` items[].media[] (JSONB 문서, `artifact/trips-data-schema.md`). 별도 trip_media 테이블 **없음**.
- **지도 대표**: `group_region_media` (지역당 1개, group·trip FK).

```sql
CREATE TABLE group_region_media (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  group_id    BIGINT       NOT NULL REFERENCES groups(group_id),
  trip_id     BIGINT       REFERENCES trips(trip_id) ON DELETE SET NULL,  -- 출처 보존
  sido_code   INT          NOT NULL REFERENCES sidos(sido_code),
  gugun_code  INT,                                                         -- nullable(시도 단독)
  media_type  VARCHAR(10)  NOT NULL CHECK (media_type IN ('PHOTO','AUDIO','VIDEO')),
  media_url   TEXT         NOT NULL,                  -- data에서 골라 복사
  geom        GEOMETRY(Point, 4326),                  -- 핀(GIST)
  metadata    JSONB        NOT NULL DEFAULT '{}'::jsonb,
  UNIQUE (group_id, sido_code, COALESCE(gugun_code, -1))   -- 지역당 1개
);
CREATE INDEX idx_grm_group      ON group_region_media(group_id);
CREATE INDEX idx_grm_geom_gist  ON group_region_media USING GIST (geom);  -- NFR-10
CREATE INDEX idx_trips_data_gin ON trips USING GIN (data);                -- NFR-09(항목/미디어 검색)
```
> ⚠️ 정본 DDL은 `backend/src/main/resources/db/postgres/schema.sql`. 위는 명세 요약.

**`items[].media`·`metadata` JSONB 예시 (타입별 자유 스키마)**
```jsonc
// PHOTO
{ "type":"PHOTO", "url":"...", "metadata":{ "w":4032, "h":3024, "thumbnailUrl":"..." } }
// AUDIO
{ "type":"AUDIO", "url":"...", "metadata":{ "durationSec":42, "format":"m4a" } }
// VIDEO
{ "type":"VIDEO", "url":"...", "metadata":{ "durationSec":120, "thumbnailUrl":"..." } }
```

### 7.6 예외
- 비멤버 업로드/조회 → 403 / 허용외 타입·포맷 → 400 / 용량 초과 → 413 / 진행 전 여행 → 미노출

### 7.7 단계 구현 권장
1. 사진 업로드·조회 + 권역 색칠 → 2. 음성 추가 → 3. 영상 + 썸네일/트랜스코딩
