## 0. 공통 사항

### 0.1 인증
- OAuth2 소셜 로그인(Google/Kakao) → JWT 발급
  - Access 20분 (`Authorization: Bearer`), Refresh 14일
- 보호 API는 `JwtAuthenticationFilter` 검증, 실패 401.

### 0.2 공통 응답 포맷
```jsonc
{ "success": true,  "data": { /* ... */ }, "error": null }
{ "success": false, "data": null, "error": { "code": "STRING", "message": "..." } }
```

### 0.3 공통 예외 코드
| HTTP | code | 설명 |
|------|------|------|
| 400 | `INVALID_PARAMETER` | 검증 실패 |
| 401 | `UNAUTHORIZED` | 인증 누락/만료 |
| 403 | `FORBIDDEN` | 권한 없음(모임 비소속 등) |
| 404 | `NOT_FOUND` | 리소스 없음 |
| 409 | `CONFLICT` | 무결성 위반(XOR, 중복) |
| 413 | `PAYLOAD_TOO_LARGE` | 업로드 용량 초과 |
| 500 | `INTERNAL_ERROR` | 서버 오류 |

### 0.4 화면 ↔ 기능 매핑 (wireframe 7화면)
| # | 화면 | URL | 기능 |
|---|------|-----|------|
| S1 | 랜딩 | `/` | 서비스 소개, 시작하기 |
| S2 | 로그인 | `/login` | 소셜 로그인 |
| S3 | 관광지 검색·목록 | `/attractions` | 지역/키워드 검색 |
| S4 | 관광지 상세 | `/attractions/{id}` | 상세·여행 추가 |

## 1. 소셜 로그인 (S1, S2 · FR-AUTH) — *기존 구현 기반*

### 1.1 개요
구글/카카오 OAuth2로 로그인하고 JWT를 발급받는다. 최초 로그인 시 회원 자동 생성. (현재 `SecurityConfig`, `CustomOAuth2UserService`, `OAuth2SuccessHandler`, `JwtProvider` 구현됨)

### 1.2 화면
- **S1 랜딩**: Hero + 3카드(자동 일정/공동 편집/관광지 검색) + "시작하기"
- **S2 로그인**: "구글로 시작하기", "카카오로 시작하기" (※ 와이어프레임의 이메일/비번 폼은 폐기 — CON-02)

### 1.3 흐름
```
[소셜 버튼] → OAuth2 동의 → CustomOAuth2UserService(회원 조회/생성)
  → OAuth2SuccessHandler(JWT 발급) → 프론트 토큰 전달 → 인증 완료 → S3 이동
```

### 1.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/oauth2/authorization/{google\|kakao}` | 소셜 로그인 시작 | - |
| GET | `/login/oauth2/code/{provider}` | 콜백(내부) | - |
| POST | `/api/auth/refresh` | Access 재발급 | refresh |
| POST | `/api/auth/logout` | 로그아웃 | ✅ |
| GET | `/api/auth/me` | 내 정보 | ✅ |

### 1.5 데이터 (`users`)
| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | BIGINT PK | |
| email | VARCHAR UNIQUE | |
| name | VARCHAR | |
| profile_image_url | VARCHAR | |
| role | VARCHAR | ADMIN/USER |
| created_at | DATE | |

### 1.6 예외
- 이메일 미동의 → 400 / 토큰 만료 → 401(→refresh) / refresh 무효 → 재로그인

## 4. 여행 계획 자동 생성 (S5 · FR-PLAN-01,02) ⭐

### 4.1 개요
지역/기간/인원/**스타일**을 입력하면 TourAPI 관광지를 기반으로 날짜·시간별 일정을 자동 추천한다. "스타일"은 콘텐츠 타입 선호로 매핑(CON-05): 예) 자연→관광지, 맛집→음식점, 문화→문화시설.

### 4.2 화면
- 입력 폼: 지역(시/도, 구/군), 출발일·종료일, 인원, **여행 스타일(다중 선택)**
- "자동 생성 ✨" → 결과는 S6(편집 화면)으로 이동

### 4.3 자동 생성 로직(초안)
```
입력: region, startDate, endDate, headcount, styles[]
1) 후보 = attractions WHERE 지역 일치 AND contentTypeId ∈ map(styles)
2) 정렬 = (이미지 유무, 대표성, 콘텐츠타입 매칭도)
3) 일자 분배 = 기간 일수만큼 Day 분할, Day당 N개(예: 3~4) 배치
4) 시간 슬롯 = 오전/점심(음식점 우선)/오후/저녁 규칙 배치
5) trips + trip_attractions(visit_order, visit_date) 생성
```
> 알고리즘 고도화(동선 최적화·거리 기반)는 후속. MVP는 규칙 기반.

### 4.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| POST | `/api/plans/auto` | 조건 입력→일정 자동 생성 | ✅ |

**요청 예**
```jsonc
{ "sidoCode":1, "gugunCode":1, "startDate":"2026-06-01", "endDate":"2026-06-03",
  "headcount":2, "styles":["자연","도보"], "groupId": 10 }   // groupId 없으면 개인
```

### 4.5 데이터 & 트리거
- 저장: `trips`, `trip_attractions`
- 추가 시 `trip_region_snapshots` upsert(`attraction_count += 1`)

### 4.6 예외
- 후보 부족 → 가능한 만큼 생성 + `partial:true` 안내
- endDate<startDate → 400 / 모임 비소속 groupId → 403 / XOR 위반 → 409

## 5. 여행 계획 편집 + 실시간 공동 편집 (S6 · FR-PLAN-03~05, FR-COL) ⭐

### 5.1 개요
일자별 일정을 조회·편집하고, **모임원이 동시에 실시간 공동 편집**한다. 편집은 WebSocket으로 송신, **Redis Pub/Sub로 서버 간 fan-out**되어 모든 참가자에게 즉시 전파된다(수평 확장 지원).

### 5.2 화면
- 헤더: 여행 제목/기간/인원 + **참가자 아바타** + 🔗공유 + ⋯
- 본문: Day별 타임라인(시간·관광지), 항목별 "○○ 편집중…" 표시, "+ 일정 추가"
- 우하단: 🟢 실시간 동기화 상태

### 5.3 REST API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/api/trips/{tripId}` | 여행 상세(일정 포함) | ✅ 멤버 |
| PATCH | `/api/trips/{tripId}` | 제목/기간 수정 | ✅ 멤버 |
| DELETE | `/api/trips/{tripId}` | 여행 삭제 | ✅ 소유/멤버 |
| POST | `/api/trips/{tripId}/attractions` | 항목 추가 | ✅ 멤버 |
| PATCH | `/api/trips/{tripId}/attractions/{id}` | 순서/시간/날짜 변경 | ✅ 멤버 |
| DELETE | `/api/trips/{tripId}/attractions/{id}` | 항목 삭제 | ✅ 멤버 |

### 5.4 실시간 채널 (WebSocket + Redis)
| 종류 | 경로 | 설명 |
|------|------|------|
| WS connect | `/ws/plans/{tripId}` | 편집 세션 입장(인증·멤버 검증) |
| 송신 이벤트 | `EDIT_ADD / EDIT_UPDATE / EDIT_DELETE / EDIT_REORDER / PRESENCE` | 편집 동작 |
| Redis Pub/Sub | `plan:{tripId}:edit` | 서버 간 fan-out |
| Redis(상태) | `plan:{tripId}:state` | 현재 일정 스냅샷(신규 참가자 즉시 로드) |

**동기화 흐름 (아키텍처 다이어그램 ① ~ ⑥)**
```
편집 → ① 서버가 Redis에 상태/이벤트 쓰기 → ② PUBLISH plan:{id}:edit
     → ③ 각 서버 SUBSCRIBE → 연결된 클라이언트로 push
신규 참가 → ④ Redis state 조회로 현재 일정 즉시 로드
Batch Worker → ⑤ Pull → ⑥ trip_attractions 영속화(정합성 보정)
```

### 5.5 동시 편집 정책(MVP)
- 항목 단위 낙관적 갱신 + last-write-wins + 버전 필드로 충돌 감지
- 고급 CRDT/OT는 후속

### 5.6 예외
- 멤버 아님 → 403 / 삭제된 항목 편집 → 409 `CONFLICT` / 연결 끊김 → 재연결 후 state 재동기화

## 7. 멀티미디어 발자취 지도 (S8 · FR-MAP) ⭐ 핵심

### 7.1 개요
그룹이 **다녀온(end_date<오늘)** 여행의 관광지 위치에, 그룹원이 남긴 **사진·음성·영상**을 지도 마커로 누적 표시한다. 권역 누적 색칠은 `trip_region_snapshots` 그룹 집계, 미디어는 `trip_media`로 제공.

### 7.2 화면
1. **발자취 지도**: 그룹 방문 시도/구군 누적 색칠(choropleth) — 진할수록 많이 방문
2. **드릴다운**: 권역 클릭 → 관광지 마커
3. **미디어 팝업**: 마커 클릭 → 사진 갤러리 / 음성 플레이어 / 영상 플레이어
4. **업로드**: 여행 상세에서 관광지별 미디어 업로드(사진 우선 → 음성 → 영상)

### 7.3 흐름
```
group_id
 → trips(group_id, end_date < today)
   → trip_region_snapshots GROUP BY sido,gugun   // 권역 누적(색칠)
   → trip_media(content_id 위치)                 // 마커 미디어
```

### 7.4 API
| Method | Path | 설명 | 인증 |
|--------|------|------|------|
| GET | `/api/groups/{groupId}/footprint` | 방문 권역 누적 집계 | ✅ 멤버 |
| GET | `/api/groups/{groupId}/media?sidoCode=&gugunCode=&contentId=` | 권역/관광지별 미디어 | ✅ 멤버 |
| POST | `/api/trips/{tripId}/attractions/{contentId}/media` | 미디어 업로드(멀티파트) | ✅ 멤버 |
| DELETE | `/api/media/{mediaId}` | 미디어 삭제 | ✅ 작성자/관리 |

**응답 예 (`/footprint`)**
```jsonc
{ "success": true, "data": [
  { "sidoCode":26, "gugunCode":1, "visitCount":3, "attractionCount":14, "mediaCount":27 }
], "error": null }
```

**업로드 (`multipart/form-data`)**: `file`(바이너리), `mediaType`(PHOTO|AUDIO|VIDEO), `metadata`(JSON, 선택)
→ 서버: 스토리지 업로드 → URL/썸네일 → `trip_media` insert

### 7.5 데이터 — `trip_media` (PostgreSQL, JSONB 하이브리드)
```sql
CREATE TABLE trip_media (
  media_id    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  trip_id     BIGINT       NOT NULL REFERENCES trips(trip_id),
  content_id  INT          REFERENCES attractions(content_id),  -- 지도 좌표 연결
  user_id     BIGINT       NOT NULL REFERENCES users(id),
  media_type  VARCHAR(10)  NOT NULL CHECK (media_type IN ('PHOTO','AUDIO','VIDEO')),
  media_url   VARCHAR(500) NOT NULL,
  geo         GEOGRAPHY(Point, 4326),                  -- (선택) 촬영 좌표
  metadata    JSONB        NOT NULL DEFAULT '{}'::jsonb, -- 타입별 유연 메타
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_media_trip     ON trip_media(trip_id);
CREATE INDEX idx_media_content  ON trip_media(content_id);
CREATE INDEX idx_media_meta_gin ON trip_media USING GIN  (metadata);  -- NFR-09
CREATE INDEX idx_media_geo_gist ON trip_media USING GIST (geo);       -- NFR-10
```

**`metadata` JSONB 예시 (타입별 자유 스키마)**
```jsonc
// PHOTO
{ "width":4032, "height":3024, "thumbnailUrl":"...", "exif":{ "takenAt":"...","gps":[35.1,129.0] } }
// AUDIO
{ "durationSec":42, "format":"m4a", "transcript":"파도 소리..." }
// VIDEO
{ "durationSec":120, "codec":"h264", "resolution":"1080p", "thumbnailUrl":"..." }
```

### 7.6 예외
- 비멤버 업로드/조회 → 403 / 허용외 타입·포맷 → 400 / 용량 초과 → 413 / 진행 전 여행 → 미노출

### 7.7 단계 구현 권장
1. 사진 업로드·조회 + 권역 색칠 → 2. 음성 추가 → 3. 영상 + 썸네일/트랜스코딩
