## 슬라이드 3 — 기획 배경 · 목표

### 문제 (Pain)
- 여행 계획은 **정보 탐색 → 동선 짜기 → 일행과 합의**까지 시간이 많이 든다.
- 단톡방·메모장·지도를 오가며 짜다 보면 **버전이 흩어지고**, 누가 뭘 바꿨는지 모른다.
- 다녀온 뒤 사진·기록은 갤러리에 흩어져 **여행의 추억이 한곳에 모이지 않는다.**

### 목표 (3단계 라이프사이클로 해결)
| 단계 | 슬로건 | 핵심 기능 |
|------|--------|-----------|
| **계획** | 빠르게 | AI 여행 자동 생성 (지역·기간·스타일 → 일정) |
| **공유** | 함께 | 실시간 공동 편집 (Google Docs 스타일) |
| **기억** | 남기다 | 멀티미디어 발자취 지도 (지역별 대표 추억) |

> **추가 기능 중심**: 단순 CRUD를 넘어 ①AI 자동생성 ②실시간 협업 ③공간 데이터 기반 발자취 지도 3종을 핵심 차별 기능으로 격상.

## 슬라이드 5 — 시장 분석 · 차별화 전략

### 경쟁 서비스 비교
| 서비스 | AI 일정 자동생성 | 실시간 공동편집 | 추억 기록(발자취 지도) | 비고 |
|--------|:---:|:---:|:---:|------|
| 트리플(Triple) | △ 추천 위주 | ✕ | △ 타임라인 | 일정 추천 강자, 협업 약함 |
| 마이리얼트립 | ✕ | ✕ | ✕ | 상품 예약 중심 |
| 구글맵 타임라인 | ✕ | ✕ | △ 자동 기록 | 계획 도구 아님 |
| 네이버 여행 | △ | ✕ | ✕ | 정보 검색 중심 |
| **어디갈래?(WSWG)** | **✅ 의미검색+GPT** | **✅ WebSocket** | **✅ 공간 데이터** | 3단계 통합 |

### 차별화 포인트
1. **계획부터 추억까지 하나의 흐름** — 경쟁사는 예약/검색/추천 중 한 조각만 담당.
2. **진짜 동시 편집** — 단톡 공유가 아니라 같은 문서를 함께 고치는 실시간 협업.
3. **AI 의미 검색 추천** — 키워드 매칭이 아닌 임베딩(pgvector) 기반 "분위기" 추천.
4. **공간 데이터 발자취 지도** — PostGIS로 지역별 방문·대표 추억을 지도에 색칠.

## 슬라이드 6 — 개발 결과 (핵심 기술·구현)

### 핵심 기능 4종
1. **AI 여행 자동 생성** — 3단계 파이프라인
   `사용자 조건 → ① GPT 후보 생성 → ② 임베딩 의미검색 매칭(pgvector) → ③ 이동거리·동선 확정 → trips.data`
2. **실시간 공동 편집** — WebSocket + Redis Stream 으로 편집 이벤트 브로드캐스트, Batch Worker가 `trips.data`에 주기 flush, 프레즌스(접속자) 표시.
3. **멀티미디어 발자취 지도** — 사진/음성/영상 업로드 → 지역별 대표 추억 1개를 PostGIS 좌표 핀 + 권역 색칠(choropleth)로 시각화.
4. **관광 데이터 파이프라인** — 한국관광공사 TourAPI 동기 적재(관리자/주 1회) + 상세 write-through 캐시.

### 규모
- 백엔드 Java 클래스 **126개** (8 Controller · 16 Service · 7 DAO)
- DB 테이블 **12개** (PostGIS 공간 + pgvector 임베딩)
- 프론트 화면 **10종** (S1~S9 + 관리자)

### 시스템 구조도 (개념)
```
[Vue SPA] ──REST──> [Spring Boot]
    │  └──WebSocket──> [PlanEdit Handler] ──> [Redis Stream] ──> [Flush Worker] ──> PostgreSQL(trips.data)
    │                       │
    └── Naver Maps          ├──> [PostgreSQL + PostGIS + pgvector]
                            ├──> TourAPI / OpenAI / Naver / ODsay (외부)
```

## 슬라이드 9 — 적용 패턴 · 핵심 알고리즘

### 적용 패턴
- **계층형 아키텍처** Controller → Service → DAO(MyBatis) → DB
- **하이브리드 데이터 모델** — 관계가 있는 것은 테이블, 자기 안에서 끝나는 항목·미디어는 `trips.data` **JSONB 문서**
- **Write-through 캐시** — 관광지 상세를 최초 조회 시 TourAPI에서 받아 DB에 채우고 재사용
- **Event Sourcing 유사 흐름** — 편집 이벤트를 Redis Stream에 적재 → 비동기 flush

### 핵심 알고리즘
1. **AI 추천**: 사용자 조건 임베딩 → `pgvector` HNSW 코사인 유사도로 관광지 매칭 → GPT가 일자별 동선 구성.
2. **이동거리/시간**: 자동차=Naver Directions, 대중교통=ODsay 로 구간 계산 → 일정 현실성 보정.
3. **공간 집계**: PostGIS `geom`(GENERATED) + GiST 인덱스로 지역별 방문 집계·근접 검색.
4. **동시성**: 지역당 대표 추억 1개 UNIQUE 제약 + XOR 제약(개인/모임 여행 배타)으로 DB 레벨 무결성 보장.

## 슬라이드 10 — 기대 효과

- **계획 시간 단축**: 정보 탐색·동선 구성을 AI가 대신 → 초안 작성 시간 대폭 감소.
- **합의 비용 감소**: 실시간 공동 편집으로 단톡 왕복·버전 충돌 제거.
- **여행 경험 완결성**: 계획→공유→기억까지 한 서비스에서 → 사용자 리텐션 상승 기대.
- **확장성**: TourAPI·임베딩·공간 데이터 파이프라인은 타 지역/도메인으로 확장 가능.
