Redis가 분산의 핵심
  • 편집 Stream — 모든 서버가 같은 Redis Stream에 편집 append
  • Pub/Sub fan-out — A 서버 편집을 B 서버의 WebSocket 세션까지 브로드캐스트
  • 분산 Lock — plan:{id}:flushLock 으로 여행당 단 1대만 DB flush (정확히 한 번)
  • 무상태 앱 — 세션·편집 상태를 서버에 두지 않아 A/B 어느 쪽이 받아도 동일 동작

TourAPI 주간 자동 적재

@Scheduled(cron="0 0 4 * * MON") — 매주 월요일 오전 4시 자동 실행. 변경분만 반영.

1

전체 목록 수집 (areaBasedSyncList)

page1로 totalCount 확인 → 페이지 루프(numOfRows)로 전국 관광지 수집

2

검증 · 필터

sido·gugun·contentType FK 유효성 검사, 좌표·필수값 누락은 skip (skippedValidation/FK 집계)

3

Upsert (멱등)

ON CONFLICT (content_id) DO UPDATE — content_id 기준으로 신규는 INSERT, 기존은 갱신

4

감사 로그

batch_run_log에 1행 기록 — SUCCESS · DEGRADED · ABORTED

🔍 변경 감지 = modified_time
  • content_id = TourAPI 고유키 → 같은 장소는 항상 같은 행에 매핑
  • 응답의 modifiedtime을 함께 저장 → "언제 바뀌었는지" 추적
  • Upsert가 변경분을 덮어쓰기하므로 별도 diff 불필요 (멱등)
  • 상세(A-6)는 modified_time이 바뀌면 write-through 캐시 무효화 → 재조회

관리자 수동 적재(POST /load)로도 동일 엔진 실행 가능

AI 벡터 검색 · 코사인 유사도

키워드 매칭이 아닌 "의미(분위기)" 검색 — OpenAI 임베딩 + pgvector.

1

임베딩 생성

관광지 설명을 text-embedding-3-small로 1536차원 벡터화 → attraction_embeddings (HNSW 인덱스)

2

질의 벡터화

사용자 조건("바다 보이는 한적한 카페")도 같은 모델로 임베딩

3

코사인 유사도 정렬

pgvector <=> 연산자로 최근접 K개 검색 (HNSW 근사)

similarity = 1 − (e.embedding <=> query)
<=> = 코사인 거리 (0=동일, 2=정반대) · 1−거리 = 유사도
-- 의미상 가까운 관광지 K개
SELECT a.title,
  1 - (e.embedding <=> :q) AS similarity
FROM attractions a
JOIN attraction_embeddings e USING(content_id)
ORDER BY e.embedding <=> :q -- HNSW
LIMIT :k;

AI 자동 생성도 이 검색으로 후보 관광지를 매칭 → GPT가 동선을 구성

실시간 동기화 — Redis ↔ DB 디바운스

키 입력마다 DB를 쓰지 않는다. 편집은 Redis에 쌓고, 10초 주기로 한 번에 flush.

편집 이벤트 (ADD / UPDATE / DELETE / 타이핑)
│ 즉시 append  ·  다른 편집자에게 Pub/Sub 브로드캐스트
Redis Stream plan:{tripId}:stream — 라이브 편집 누적 (빠른 쓰기)
│ @Scheduled(fixedDelay = 10s) · 디바운스
Flush Worker 분산 Lock 획득 → lastFlushedId 이후 레코드만 → trips.data 반영
│ 통째 1회 쓰기
PostgreSQL · trips.data (JSONB)
왜 디바운스인가
  • DB 쓰기 폭증 방지 — 키 입력당 1 write → 10초당 1 write
  • 응답 즉시성 — 화면 반영은 Redis로 즉시, DB는 비동기
  • 정확히 한 번 — flushLock + lastFlushedId 커서로 여러 서버 중복 flush·재처리 방지
  • 유실 안전 — flush 전 장애 시 Stream에 편집이 남아 다음 주기에 복구

트러블슈팅 ① MySQL → PostgreSQL 전환

왜 바꿨나
  • 공간 데이터 — 발자취 지도·근접 검색에 GIS가 필요한데 MySQL은 기능·인덱스가 제한적

  • 둘을 한 DB에서 → PostgreSQL + PostGIS + pgvector로 결정
전환하며 바뀐 것
  • 확장 PostGIS(geom·GiST) · pgvector(HNSW) 도입
  • 모델 단순화 11→9 테이블 (항목·미디어를 trips.data JSONB로 통합)
  • 문법 upsert를 ON CONFLICT ... DO UPDATE 로 전환
  • 배포 교훈 설정은 ${REDIS_HOST:localhost} env-fallback 으로 — 하드코딩 host가 운영 redis를 덮어써 크래시 → 수정

05Troubleshooting

트러블슈팅 ② 실시간 편집 — 메시지 유실

증상: 빠르게 타이핑하면 일부 글자가 저장되지 않거나 되돌아감.

🐞 원인 A · PUT 폴백이 Redis 우회
편집 중 REST PUT이 Redis state를 건너뛰어, flush 워커가 옛 상태로 DB를 덮어씀 → 입력이 되돌아감
🐞 원인 B · 편집 락 안에서 DB 경합
한 락 안에서 DB를 두 번 읽어 경합 → BUSY 거부를 FE가 조용히 폐기
🐞 원인 D · 재연결 sync 덮어쓰기
재접속 시 서버 state가 로컬 미전송 편집을 덮어씀

✅ 해결 — FE 측

편집을 op 큐(clientOpId)에 적재 → ack 받으면 제거 · BUSY는 백오프 재시도 · ack 타임아웃(4s) 재전송 · sync 수신 시 미전송 op 재적용 (PUT 폴백 제거)

✅ 해결 — BE 측

권한 확인·state 워밍을 락 밖으로 (락 안은 Redis 전용) · appliedOps 집합으로 멱등 dedup (재전송 중복 적용 차단) · 제목·날짜는 PATCH /meta 로 분리

검증: BE 8 + FE 21 테스트 통과 · 텍스트 필드 last-writer-wins는 설계 한계로 후순위
